## 1. 日タブ: ミニカレンダーを前後含む3週表示に

**現状**: `src/components/sidebar/MiniCalendar.tsx` の compact モード（`calendarMode === "day"` のとき）は
`startOfWeek(anchorDate)` を先頭に **現在週から先の1〜2週だけ**を表示している
（`compactWeeks = windowHeight >= 760 ? 2 : 1`、`days = Array.from({length: compactWeeks*7}, (_,i) => addDays(weekStart, i))`）。
このため「次」側の日をクリックして anchorDate が翌週に移ると、前の週がリストから消えて戻れなくなる。

**修正方針**:
- compact モードの表示を「**前週・当週・翌週の3週固定**」に変更する。
  `days = Array.from({ length: 21 }, (_, i) => addDays(addDays(weekStart, -7), i))`
- `compactWeeks` / `TWO_WEEK_MIN_WINDOW_HEIGHT` / `windowHeight` の resize 監視は不要になるため削除
  （`noUnusedLocals` が有効なので、消し忘れるとビルドが落ちる）
- 当週（anchorDate を含む週）のハイライト（`in-week` クラス）は現状ロジックのままでよい。
  前後週の日も従来どおりクリックで `setAnchorDate(d)` により移動できる（これが「戻れる」手段になる）

**確認**: 日タブでミニカレンダーに3週分が出ること。翌週の日をクリック後も前週が見えていて戻れること。

## 2. 日タブ: デフォルト表示を9:00–20:00程度に・行の情報量を増やす

**現状**: `src/components/calendar/DayView.tsx` は
`MIN_PX_PER_MIN = 960 / MINUTES_PER_DAY`（=0.667px/分）を下限に、
**トラック実測幅から「24時間が横スクロールなしで収まる」px/分を算出**している
（`el.clientWidth / MINUTES_PER_DAY` を ResizeObserver で監視）。
つまり常に0時〜23時が1画面に押し込まれ、1コマが極端に狭い。
`scrollTo(scheduleStartHour * 60 * pxPerMin, 0)` は既にあるが、スクロール余地がないため実質無効。

**修正方針**:
- px/分の算出基準を「24時間で幅いっぱい」から「**約11時間（9:00–20:00相当）で幅いっぱい**」に変更する。
  定数 `VISIBLE_HOURS = 11` を導入し、`perMin = el.clientWidth / (VISIBLE_HOURS * 60)` とする
  （トラック全体の幅は `MINUTES_PER_DAY * pxPerMin` になり、自然に横スクロールが発生する）。
- 既存の `scrollTo(scheduleStartHour * 60 * pxPerMin, 0)` が生きてくるので、
  初期表示は `scheduleStartHour`（既定9時、設定変更可）から始まる。この動作を実機で確認すること。
- 「同じ時間に2つ記録することは基本ない」ため、エントリブロック内の情報量を増やす:
  1コマが広くなるので、`EntryBlock` に渡している表示内容（現状タイトル+時刻）に加えて
  **メモの先頭1行とカテゴリ名**を、日タブのブロック内にだけ表示する
  （`EntryBlock.tsx` に optional prop `detail?: string` などを追加し、DayView からのみ渡す。
  WeekView の表示は変えない）。行高 `ROW_HEIGHT = 46` は、詳細表示を入れるなら 60 程度へ調整可。
  逆に重なりが無い日は `rowCount = 1` なのでトラック全体の縦幅は元々小さい。
  「縦幅を縮小するか、詳細も表示するか」の要望に対しては**詳細表示を採る**（縦幅は既に最小に近いため）。

**確認**: 日タブ初期表示が9時始まりで、およそ20時までが見えること。左右スクロールで0時〜23時に到達できること。
エントリにメモ・カテゴリが表示されること。ドラッグでの新規作成/移動/リサイズが引き続き正しい時刻にスナップすること
（`pointToMin` は pxPerMin ベースなので自動で追従するはずだが、実際にドラッグして確認）。

## 3. 日タブ: ウィンドウ最小化とサイドバーの手動開閉

**現状**:
- `src/App.tsx` に「calendarMode が day になったら Tauri Window API でウィンドウを 960×(元の高さ×0.75) に
  リサイズし、day を抜けたら元に戻す」処理が実装済み（`LogicalSize` 使用）。
- サイドバー（ミニカレンダー・カテゴリ等の列）は、幅ベースの自動非表示に加えて
  `appStore.sidebarManuallyHidden` + Toolbar の `IconSidebar` トグルボタンが**実装済み**。

**残作業（要検証・調整）**:
- ユーザーは「まだ画面全体サイズが最小になっていない」と感じている。実機（`npm run tauri dev`）で
  day タブに切り替えた際に実際にウィンドウが縮むかを確認する。
  縮まない場合は App.tsx のリサイズ effect の発火条件（view が "calendar" かつ calendarMode が "day"）と、
  `getCurrentWindow().setSize()` の呼び出しが失敗していないか（Promise の reject を握りつぶしていないか）を調べて直す。
- 項目2で日タブの必要幅が「11時間分」に減るため、day モード時の目標幅を 960px から
  さらに小さくできるかを検討する。ただし `src-tauri/tauri.conf.json` の `minWidth: 960` が下限のため、
  **960 未満にしたい場合は `minWidth` を 720 程度に下げる**（週タブの最小幅827px+αを考慮しつつ、
  day モード時だけ小さくできるよう `setMinSize` を動的に切り替える方法もある。
  過剰に複雑にせず、まずは `minWidth: 720` に下げて day 時 720×0.75h にするのが簡便）。
- サイドバー手動トグルが day タブでも機能すること（Toolbar は calendar ビュー共通なので動くはず）を確認。

**確認**: 日タブに切り替えるとウィンドウが実際に縮小し、週に戻すと元サイズへ復元。
トグルボタンでサイドバーの表示/非表示が切り替わる。

## 4. チケット: 選択/非選択で表示がずれる問題の再修正

**現状**: 前回の修正で「詳細パネル（`.tickets-tab-detail`、`min-width: 320px`）を常時レンダリングし、
未選択時はプレースホルダを出す」対応済みのはずだが、**まだ微妙にずれる**とのフィードバック。

**調査ポイント（実装前に必ず特定すること）**:
- `.tickets-tab-master` は `flex: 1.3`、`.tickets-tab-detail` は `flex: 1; min-width: 320px`。
  プレースホルダ（`.ticket-detail-empty`）と実際の詳細（`.ticket-detail`）で
  **border / padding / 内容幅が異なり、flex 配分が数px変わる**可能性が高い。
  `.ticket-detail` は `border: 1px solid var(--border); padding: 16px` を持つ。
  プレースホルダ側が同じ border/padding を持っているか確認し、無ければ合わせる。
- もう1つの原因候補: 選択時にのみ現れる**縦スクロールバー**（詳細パネルや作業ログの max-height 由来）が
  レイアウト幅を変えるパターン。`scrollbar-gutter: stable` を `.tickets-tab-detail` / `.tasks-view` に
  付与して検証する。
- DevTools が使えない環境のため、`getBoundingClientRect()` をコンソール等から測るのではなく、
  選択/非選択の両状態で同じ CSS ボックス（border+padding+width 固定）になるよう静的に揃えるのが確実。

**修正方針**: プレースホルダを `.ticket-detail` と**同一のクラス構成**でレンダリングし
（visibility だけ変えるか、同じ枠内に空メッセージを置く）、`flex` 依存の幅差を無くす。
必要なら `.tickets-tab-detail` を `flex: 1` から `width: 380px; flex: none` の固定幅に変更してよい
（マスター側が `flex: 1` で残りを取る構成の方がずれようがない）。

## 5. チケット: 状態・カテゴリのバッジを中央揃え・幅適正化

**現状**: チケット一覧の行は `src/components/tasks/TicketRow.tsx` + `styles.css` の
`.ticket-row-grid { grid-template-columns: 76px 1fr 60px 110px 64px 64px }`。
- 状態バッジ `.lozenge` は `padding: 2px 8px` のインライン要素で、**グリッドセル（76px）内で左寄せ**。
  背景がテキスト幅にしかならず不揃いに見える or セルによっては間延びして見える。
- カテゴリ `.ticket-row-category` も同様に 110px セル内で左寄せ・可変幅。

**修正方針**（カンバン側は前回対応済み。今回は**チケット一覧の行**が対象）:
- `.lozenge`（一覧内のもの）: 幅をセル内で固定見た目にする。
  `display: inline-flex; justify-content: center; width: 64px;`（状態名は最長「進行中」3文字なので収まる）
  ＋ `text-align: center`。グリッド1列目 76px の中でバッジ幅を統一し中央に置く
  （セル側に `justify-self: center` でも可）。
- `.ticket-row-category`: 同様に `display: inline-flex; justify-content: center; max-width: 100%;`
  ＋ セル内中央寄せ（`justify-self: center`）。padding は `2px 8px` 維持、
  背景がテキストに追従する丸タグのまま**「長すぎない」よう max-width をセル幅(110px)以内に**。
- カンバン `.kanban-status-chip` は `max-width: 220px` が既にあるが、
  「長すぎる」再指摘に備え `width: fit-content; margin: 0 auto;`（カード内中央）へ調整し、
  padding を `2px 12px` 程度に抑える。
- ヘッダー行 `.ticket-list-header` のラベル位置も、上記の中央寄せに合わせて
  1列目・4列目を `text-align: center` にする。

## 10. アプリ・タブのアイコン刷新（Claude Code 風）

**対象**: (a) NavRail のタブアイコン（`src/components/icons.tsx`）、(b) アプリ本体のアイコン

塗りなし、単色 currentColor）。現行 `icons.tsx` は既にこの流儀

- カレンダー / チケット(チェック付きリスト) / 分析(棒グラフ) / ノート / 履歴(時計+矢印) /
  設定 / サイドバートグル──を同一グリッド(24px, 2pxストローク相当)で統一感を持って描き直す。
  すべて `icons.tsx` 内の SVG パス差し替えで完結し、コンポーネント名・props は変えない。
- アプリアイコン: 「時間を塗るキャンバス」を表す簡潔なマーク
  （角丸正方形の背景 + 時計の針 or カレンダーマスのモチーフ、アクセント色 #2383e2）を
  SVG で作成 → 1024px PNG に書き出し → `npm run tauri icon <png>` で
  `src-tauri/icons/` 一式（.ico/.png 各サイズ）を自動生成する。
  SVG→PNG 変換は WSL に ImageMagick が入っている（`convert`）のでそれを使う。
- 生成後 `tauri.conf.json` の `bundle.icon` は既存パス参照のままでよい（ファイルが置き換わるだけ）。

**確認**: `npm run tauri dev` でウィンドウ/タスクバーのアイコンが変わること。NavRail の見た目確認。
