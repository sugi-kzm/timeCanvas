# TimeCanvas 要件定義書

## 1. プロジェクト概要

### 1.1 背景

Outlook のスケジュール機能は使いやすいが、記録した予定・実績が組織内の他者にも見えてしまう。
「自分が何を何時間やったのか」を**完全にローカルで、自分だけが見られる形で**記録・振り返りできる
Windows デスクトップアプリを作る。

### 1.2 プロダクト名

**TimeCanvas** — 時間をキャンバスに描くように記録する。

### 1.3 コアバリュー

| 価値 | 説明 |
|------|------|
| プライバシー | データは端末内にのみ保存。ネットワーク送信は一切行わない |
| 慣れた操作感 | Outlook カレンダーと同等の見た目・操作感で学習コストゼロ |
| 振り返り | 記録したデータを分析・年間振り返りに活用できる（将来フェーズ） |

## 2. フェーズ計画

| フェーズ | 内容 | 状態 |
|---------|------|------|
| **Phase 1 (MVP)** | 時間記録：週カレンダーにドラッグで実績を記入・編集・削除。アプリシェル（画面切替の骨組み）と OneDrive 自動バックアップも含む | ← 今ここ |
| **Phase 2** | TODO リスト（タスク管理画面）、タスクの見積時間と実績時間の紐付け、エントリ検索 |
| **Phase 3** | 分析画面：見積 vs 実績の比較、カテゴリ別集計、年間振り返りビュー |
| **Phase 4** | ノート（ナレッジベース）：経費精算のやり方等の手順を階層構造で整理・参照 |

Phase 1 のデータモデルと画面構造は Phase 2〜4 を見据えて設計する（後述）。

## 3. Phase 1 機能要件

### 3.1 カレンダー表示

- **週表示**をデフォルトとする（Outlook の「稼働週/週」ビュー相当）
- 日表示・月表示への切り替え
- 「今日」ボタン、前後の週/日/月への移動
- 現在時刻を示すインジケーターライン
- 時間軸は 00:00〜24:00、30 分単位のグリッド

### 3.2 実績（タイムエントリ）の記録

- カレンダー上を**ドラッグして時間帯を選択 → タイトル入力**で即記録
- エントリのドラッグ移動（時間帯・日の変更）
- エントリの端をドラッグして時間の伸縮
- クリックで詳細編集（タイトル、カテゴリ、開始/終了、メモ）
- 削除（詳細画面 or 右クリック or Delete キー）
- 最小単位：15 分。スナップは 15 分刻み

### 3.3 カテゴリ

- エントリにカテゴリを 1 つ付与できる（例：開発、会議、調査、雑務）
- カテゴリごとに色を設定。カレンダー上のエントリはカテゴリ色で表示
- カテゴリの追加・編集・削除・アーカイブ

### 3.4 週サマリー

- 表示中の週の合計記録時間、カテゴリ別内訳（時間と割合）をサイドパネルに表示
- Phase 3 の分析機能の入り口となる最小限の集計

### 3.5 データ管理

- 自動保存（操作の都度 SQLite に書き込み。保存ボタンは設けない）
- JSON / CSV エクスポート（バックアップと将来のデータ活用のため）

### 3.6 バックアップと PC 引き継ぎ

ラップトップを年 1 回程度交換するため、記録を確実に引き継げることを必須要件とする。

**方式：ローカル DB + クラウドフォルダへの自動バックアップ**

- DB 本体は常にローカル（`%APPDATA%/TimeCanvas/`）に置く
- バックアップ先フォルダをユーザーが指定できる（**OneDrive 配下のフォルダを推奨既定値として案内**）
- バックアップのタイミング：アプリ終了時 + 起動中は 1 日 1 回
- バックアップ方法：SQLite の `VACUUM INTO`（安全なスナップショット）で単一ファイルを生成
- 世代管理：直近 30 世代を保持し、古いものは自動削除
- **復元**：初回起動時に「バックアップから復元」を選べる。設定画面からも復元可能
- 新 PC への引き継ぎ手順：アプリをインストール → 初回起動で OneDrive のバックアップフォルダを指定 → 復元、の 3 ステップで完了すること

> **DB 本体を OneDrive 直下に置く方式は不採用**とする。理由：
> SQLite の書き込み中に OneDrive の同期が走ると DB 破損の恐れがある（WAL の
> `-wal` / `-shm` ファイルと本体の不整合、ロック競合、競合コピーの生成）。
> また「ファイルオンデマンド」でローカル実体がない場合に起動不能になる。
> バックアップファイル（書き込みが完了した静的ファイル）のみを同期対象にすることで
> この問題を回避する。

### 3.7 アプリシェル（画面ナビゲーション）

左端に**ナビゲーションレール**（Outlook のアプリバー相当）を置き、画面を切り替える：

| アイコン | 画面 | フェーズ |
|---------|------|---------|
| カレンダー | 時間記録（メイン画面） | Phase 1 |
| チェックリスト | タスク（TODO・見積） | Phase 2 |
| グラフ | 分析（集計・年間振り返り） | Phase 3 |
| ノート | ナレッジベース | Phase 4 |
| 歯車（下端） | 設定 | Phase 1 |

Phase 1 ではカレンダーと設定のみ有効。他はレール自体を先に用意し、無効表示（またはプレースホルダー）とする。

## 4. 非機能要件

| 項目 | 要件 |
|------|------|
| プライバシー | 外部ネットワーク通信を一切行わない（アップデート確認含め Phase 1 では無し）。クラウドへ渡るのはユーザーが指定したフォルダに置くバックアップファイルのみで、同期は OneDrive クライアント任せ（アプリ自身は通信しない） |
| データ保存 | SQLite。保存先は `%APPDATA%/TimeCanvas/`（ユーザーローカル） |
| バックアップ | アプリ終了時 + 日次で指定フォルダへ自動バックアップ（3.6 参照）。失敗してもアプリ操作は妨げない（次回リトライ + 設定画面に警告表示） |
| 起動時間 | 2 秒以内に操作可能になること |
| 配布サイズ | 20MB 以下（Tauri の利点を活かす） |
| 対応 OS | Windows 10 / 11 |
| 言語 | UI は日本語 |
| データ堅牢性 | 書き込みはトランザクション単位。破損防止のため WAL モード使用 |

## 5. 技術スタック

| レイヤ | 技術 | 選定理由 |
|--------|------|---------|
| アプリ基盤 | **Tauri 2.x** | 軽量・高速・小配布サイズ。ローカル完結アプリに最適 |
| フロントエンド | **React + TypeScript + Vite** | カレンダー UI の柔軟な自作、型安全 |
| 状態管理 | Zustand（軽量）| MVP に十分。イミュータブル更新を徹底 |
| DB | **SQLite**（tauri-plugin-sql） | ローカル完結、SQL で Phase 3 の集計が容易 |
| カレンダー UI | 自作（CSS Grid ベース） | Outlook 風の見た目・操作感を細部まで制御するため |
| テスト | Vitest（unit）+ Playwright（E2E）+ Rust `cargo test` | カバレッジ 80% 以上 |

> カレンダーライブラリ（FullCalendar 等）の採用は実装開始時に再評価する。
> Outlook 風の操作感（ドラッグ記入・伸縮）が要件を満たせるなら採用してよい。

## 6. データモデル（Phase 2/3 を見据えた設計）

```sql
-- カテゴリ
CREATE TABLE categories (
  id          TEXT PRIMARY KEY,          -- UUID
  name        TEXT NOT NULL,
  color       TEXT NOT NULL,             -- HEX 例: #2564CF
  archived    INTEGER NOT NULL DEFAULT 0,
  sort_order  INTEGER NOT NULL DEFAULT 0,
  created_at  TEXT NOT NULL              -- ISO 8601
);

-- 時間エントリ（実績）
CREATE TABLE time_entries (
  id          TEXT PRIMARY KEY,          -- UUID
  title       TEXT NOT NULL,
  category_id TEXT REFERENCES categories(id),
  start_at    TEXT NOT NULL,             -- ISO 8601 (ローカルタイム + オフセット)
  end_at      TEXT NOT NULL,
  memo        TEXT NOT NULL DEFAULT '',
  task_id     TEXT,                      -- Phase 2: tasks(id) への参照（今は NULL）
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL
);

CREATE INDEX idx_entries_start ON time_entries(start_at);

-- Phase 2 で追加予定（参考）
-- CREATE TABLE tasks (
--   id, title, estimate_minutes, status, due_date, created_at, ...
-- );
```

設計意図：

- `task_id` を最初から用意し、Phase 2 で「タスク ⇔ 実績」を紐付けて見積 vs 実績分析（Phase 3）につなげる
- 日時は ISO 8601 文字列で保存し、SQLite の日付関数で年間集計が可能

## 9. 後続フェーズの詳細方針

### 9.1 Phase 2：エントリ検索の仕様

「あの作業いつやったっけ」に答える機能。記録が数ヶ月分溜まった頃に価値が出るため Phase 2 で実装する。

**検索の入り口**

- カレンダー画面ツールバー右上の検索ボックス（ショートカット：`Ctrl+F`）
- 入力するとコンテンツ領域が検索結果ビューに切り替わる。`Esc` またはクリアで元のカレンダーに戻る

**検索対象とマッチング**

- タイトル + メモを対象としたキーワード検索（部分一致）
- スペース区切りで AND 検索（例：「API 設計」→ 両方を含むエントリ）
- 大文字小文字は区別しない

**絞り込みフィルタ**

- カテゴリ（複数選択可）
- 期間：今週 / 今月 / 今年 / 全期間 / 任意の日付範囲（既定は全期間）

**検索結果の表示と動線**

- 新しい順のリスト表示：日付・時刻、カテゴリ色バー、タイトル、メモ抜粋（キーワードをハイライト）
- 件数と合計時間を表示（例：「12 件 ・ 合計 9.5 時間」→ 簡易的な「この作業に使った時間」集計を兼ねる）
- **項目クリックで該当週のカレンダーにジャンプし、そのエントリをハイライト**（前後の文脈も見える）
- ジャンプ後、`Esc` や戻るボタンで検索結果リストに復帰できる

**実装方針**

- まず SQLite の `LIKE` 検索で実装（個人利用の件数なら十分高速）
- 将来件数が増えて遅くなったら FTS5 を検討。ただし日本語は既定トークナイザで分かち書きできないため、採用時は trigram トークナイザを使う
- 性能目標：数万件のエントリに対して 100ms 以内に結果表示

### 9.2 Phase 4：ノート（ナレッジベース）の方針

「経費精算のやり方」のような手順・知識を Notion のように階層構造で整理する機能。
時間記録とは別ドメインだが、「自分だけのローカル作業ログ・知識」という本アプリの
テーマには合致するため、**独立フェーズとして最小構成で作る**。

- **保存形式は Markdown ファイル + フォルダ階層**（SQLite には入れない）
  - 階層構造がそのまま OS のフォルダで表現でき、OneDrive 同期と好相性
  - アプリが消えてもファイルとして読める（ロックインなし）
  - VS Code 等の外部エディタでも編集できる
- 機能は 3 つに絞る：①フォルダツリー表示 ②Markdown 編集/プレビュー ③全文検索
- 将来的にノート ⇔ タスク ⇔ 時間エントリのリンクを検討

## 11. リスクと対応

| リスク | 対応 |
|--------|------|
| カレンダーのドラッグ操作の実装コストが高い | まずクリック→時刻入力で作成できる最小版を作り、ドラッグは段階的に足す |
| WSL 環境での Windows 向けビルド | 開発は WSL、リリースビルドは Windows 側の Rust ツールチェーンで行う（要検証） |
| SQLite データ破損 | WAL モード + `VACUUM INTO` による自動バックアップ（3.6）で自衛 |
| OneDrive 同期による破損 | DB 本体は同期対象外（ローカルのみ）。静的なバックアップファイルだけを同期フォルダに置く |
